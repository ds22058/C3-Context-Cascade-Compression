# C3 v2.1 数据、训练与评测升级方案

## 1. 版本对比总览


| 维度  | v2.0 (当前) | v2.1 (目标) |
| --- | --------- | --------- |


- **重建数据**: v2.0 = 200万, 400-1500 tokens, 4源 → v2.1 = 200万, **3072-5120 tokens(随机)**, 7源
- **续写数据**: v2.0 = 60万, 800-2000 tokens, SlimPajama → v2.1 = 100万(训练时采样60万), 总长8192, 7源
- **latent_token_len**: v2.0 = 32 → v2.1 = 256
- **续写切分**: v2.0 = 随机0.4-0.7 ratio → v2.1 = **随机3072-5120位置切分**(前3k-5k压缩，后5k-3k生成，前后互补)
- **序列长度**: v2.0 = 变长(大量padding) → v2.1 = 小范围变长(encoder 3330-5378, decoder 3350-5398, **batch内动态padding**, 估~12%开销)

## 2. 数据来源与数量

### 重建数据 (200万, filter >=5120 tokens, 随机截断到[3072,5120])

- **FineWeb-Edu**: `HuggingFaceFW/fineweb-edu` (sample-350BT) — 80万 (40%) — **分级score筛选**(见下方)
- **SlimPajama**: `cerebras/SlimPajama-627B` — 40万 (20%) — 多领域Web
- **Proof-Pile-2**: `EleutherAI/proof-pile-2` (arxiv子集) — 30万 (15%) — 数学/科学论文
- **PG-19**: `deepmind/pg19` — 10万 (5%) — 文学/书籍, 需分chunk(每本书>>5120)
- **Wikipedia**: `wikimedia/wikipedia` (20231101.en) — 10万 (5%) — 百科
- **PubMed/arXiv**: `scientific_papers` (pubmed子集) — 15万 (7.5%) — 医学/科学
- **FreeLaw**: `pile-of-law/pile-of-law` (子集) — 15万 (7.5%) — 法律

### 续写数据 (100万, filter >=8192 tokens, 截断到8192, 随机split在[3072,5120])

**来源比例与重建一致**:

- FineWeb-Edu: 40万 | SlimPajama: 20万 | Proof-Pile-2: 15万
- PG-19: 5万 | Wikipedia: 5万 | PubMed: 7.5万 | FreeLaw: 7.5万

### FineWeb-Edu 分级质量筛选

为保证数据质量同时确保数量充足, 采用分级策略:

1. **优先收集** score >= 3.0 的样本 (高质量教育文本)
2. 如果高质量不够目标数, **降级收集** score >= 2.0 的样本作为补充
3. 日志记录各级别的采集数量, 便于追踪质量分布

实现方式: 单次stream中同时维护两个计数器。先写入高质量, 额度不足时再写入中质量:
```python
if score >= 3.0:
    write_sample()  # 总是接受
elif score >= 2.0 and high_quality_count < target * 0.9:
    write_to_backup_pool()  # 备用, 仅在高质量不足时启用
```

### 关键设计: 单次遍历同时收集两类数据

对每个来源, 一次streaming pass同时写入两个输出文件:

- text >=8192 tokens → 同时写入 recon pool 和 cont pool
- 5120 <= text < 8192 → 仅写入 recon pool
- text < 5120 → 跳过

这样只需扫描一次每个数据源, 节省一半下载时间。

### 训练代码

**`train/dataset_c3v2.py` 无需修改**。已有逻辑完全兼容:

- `latent_token_len=256` 通过参数传入
- 3k-5k text → encoder 3330-5378, decoder 3350-5398 (均远小于 max_length=8192)
- continuation 的 `split_ratio` 在 prepare 阶段按随机split点计算并写入 JSONL, dataset 直接读取
- `data_collator.py` 的 `pad_sequence` 已实现 batch 内动态 padding (pad to longest), 无需修改

### 训练优化建议: 按长度分桶减少padding

在 `prepare_data_v21.py` 中可按 context_len 排序, 使相近长度的样本在同一批次:

- 方法1: 每个 epoch 内按 context_len 排序后分桶 shuffle (桶大小=1000)
- 方法2: 在 dataset 中实现 `LengthGroupedSampler` (HuggingFace Trainer 已内置 `group_by_length` 参数)
- 效果: padding 开销从 ~12% 降至 ~2-3%

## 4. 评测集与上下文长度


| Benchmark | 类型  | Avg 上下文(words) | 估算tokens(Qwen) | Samples | 指标  | 备注  |
| --------- | --- | -------------- | -------------- | ------- | --- | --- |


- **Qasper**: Single-doc QA, avg 3619 words, ~4200-4800 tokens, 200 samples, F1 — 最接近4K训练长度; Cartridges/AM共用
- **MultiFieldQA-en**: Single-doc QA, avg 4559 words, ~5000-5900 tokens, 150 samples, F1 — 多领域, 大部分4-8K范围
- **2WikiMultihopQA**: Multi-doc QA, avg 4887 words, ~5300-6400 tokens, 200 samples, F1 — 多跳推理
- **QuALITY**: Multiple-choice QA, avg ~5000 words, ~5500-6500 tokens, ~300 samples, Accuracy — Cartridges/AM论文均用; 对比最直接
- **Reconstruction (自建)**: 文本重建, 3k-5k tokens, 500 samples, BLEU/ROUGE/exact-match — 压缩核心能力评测

注: 训练覆盖3k-5k上下文。Qasper(~4.7k tokens)落在训练分布内, MultiFieldQA/2WikiMQA/QuALITY部分样本超出5k需截断。LongBench-E的0-4K子集可完整压缩。与Cartridges/AM对比时注意: 它们支持更长上下文(16k-128k), 我们聚焦在4k-level的高压缩比(16:1-20:1)。

LongBench-E 提供了按长度均匀采样的版本, 可单独评测 0-4K 和 4-8K 区间性能:

- Qasper: 0-4K有100条, 4-8K有100条, 8K+有24条
- MultiFieldQA-en: 0-4K有67条, 4-8K有70条, 8K+仅13条
- 2WikiMultihopQA: 三个区间各100条

## 5. 时间预估

### 数据下载与处理

基于 v2.0 实际数据: 260万样本(400-2000 tokens), FineWeb-Edu扫描6.35M条/176min完成。


| 环节  | 预估时间 | 说明  |
| --- | ---- | --- |


- **数据下载**: 12-24h — FineWeb-Edu是瓶颈: >=5120 yield ~4%, 需扫描~25M条(vs v2.0的6.3M); 分级score筛选可能需要更多扫描; 7源并行但受网速限制
- **数据准备**: 15-30min — 截断 + shuffle + 写JSONL; 数据量~70GB (vs v2.0 ~3GB)
- **磁盘空间**: ~200GB — raw_v21 ~70GB + processed_v21 ~130GB

### 训练时间 (8xH200, DeepSpeed ZeRO-2)

推算依据:

- v2.0 Phase 2 实测: 23,436 steps, 133,787s (37.2h), 0.175 steps/s, MFU ~18%
- v2.0 Phase 1 实测: 17,184 steps, 87,160s (24.2h), 0.197 steps/s, MFU ~8% (短序列+冻结decoder)
- v2.1 序列增长: encoder 834→avg 4354 tokens (~5.2x), decoder 900→avg 4374 tokens (~4.9x)
- v2.1 MFU预期: 长序列GPU利用率更高, 估计20-25% (vs v2.0 Phase2的18%)

**v2.1 序列长度分布**:
- encoder: 3330-5378 tokens (avg ~4354), decoder: 3350-5398 tokens (avg ~4374)
- 重建和续写的长度范围一致 (重建target=context, 续写target=8192-context, 范围相同)
- **batch内动态padding**: 预计~12%额外开销 (启用`group_by_length`排序后可降至~3%)

- **Phase 1**: 39,062 steps, ~770 EFLOPs, **5-7天(120-170h)** — 冻结decoder; encoder(6P)+decoder(2P, forward only); 基于20% MFU; padding开销+12%
- **Phase 2**: 23,438 steps, ~840 EFLOPs, **6-8天(140-190h)** — 全模型训练; encoder+decoder 均6P; 基于20-25% MFU; 混合recon+cont的padding
- **合计**: 62,500 steps, ~1610 EFLOPs, **~11-15天**

**强烈建议**: 正式训练前先跑1000步 trial run (~1h) 实测 steps/s, 然后校准总时间。

### 与v2.0对比


| 维度  | v2.0 | v2.1 | 变化  |
| --- | ---- | ---- | --- |


- **序列长度**: ~1000 tokens → avg ~4400 tokens (range 3.3k-5.4k) — ~4.4x
- **Phase 1**: ~24h (17K steps) → ~5-7天 (39K steps) — ~6x (更多步+更长序列)
- **Phase 2**: ~37h (23K steps) → ~6-8天 (23K steps) — ~5x (同步数但长序列+padding)
- **Padding浪费**: 显著(变长400-2000) → 可控(变长3k-5k, ~12%; 启用length grouping后~3%)

### 评测时间参考

基于v2.0评测log: reconstruction ~~5min, lm_eval ~12min. 新QA评测每个benchmark估计 10-20min.
全部评测(含5个benchmark + reconstruction + lm_eval): **~~2-3小时**。
